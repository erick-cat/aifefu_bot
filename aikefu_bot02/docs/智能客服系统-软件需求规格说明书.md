## 1. 引言

### 1.1 编写目的

本文档用于明确「智能客服聊天机器人系统」的建设目标、功能边界、非功能指标及约束条件，作为产品设计、开发、测试与验收的共同依据。

### 1.2 预期读者

产品经理、架构师、后端/前端/算法工程师、测试工程师、运维及项目管理人员。

### 1.3 项目背景与目标

企业需要通过统一平台提供 7×24 小时智能应答，在自动解决常见问题的同时，支持复杂场景转人工，并对知识内容、对话质量、模型与风控进行可运营、可审计的管理。

**建设目标概要：**

- 面向终端用户：流畅对话、准确问答、业务办理入口清晰，必要时无缝衔接人工客服。
- 面向运营与管理员：知识可维护、对话可监控、模型与 Prompt 可配置、风险可控、数据可分析。

### 1.4 范围说明

**包含：**

- 用户侧账号注册与登录、会话与智能问答、业务集成入口、人工客服转接与排队体验相关能力。
- 管理侧管理员登录、知识库、监控、Prompt、模型、敏感词/风控、统计分析等模块。
- 与计划技术栈一致的分层实现：Java 主业务与数据层、Python 推理与 LangChain 编排、向量检索与嵌入服务。

**不包含（可作为后续迭代）：**

- 具体第三方业务系统（订单、支付等）的深度定制逻辑，仅以「集成规范与接口约定」形式纳入本文档；具体对接在子项目中细化。
- 非中文为主的全球化多语言（若需要可在修订版中单独成章）。

### 1.5 术语与缩写

| 术语 | 说明 |
|------|------|
| RAG | Retrieval-Augmented Generation，检索增强生成 |
| Embedding | 文本向量化，用于语义检索 |
| Prompt | 提示词模板，用于约束大模型行为与输出格式 |
| 会话 Session | 用户一次连续交互的上下文边界，通常有超时策略 |
| 兜底 | 自动应答无法解决时，转人工或给出明确指引 |
| 凭证 | 登录成功后由服务端签发的 Token 或会话标识，用于后续请求鉴权 |

### 1.6 参考依据

- 企业内部信息安全与数据合规要求（实施阶段需单独成文并引用版本号）。
- 所选大模型与 Embedding 服务商的使用条款与限流策略。

## 2. 总体描述

### 2.1 产品形态

Web 端为主：用户通过嵌入业务站点或独立 H5/PC 客服窗口访问；管理员通过独立后台进行管理。

### 2.2 用户角色

| 角色 | 描述 | 主要诉求 |
|------|------|----------|
| 访客/登录用户 | 使用客服能力的前端用户 | 快速解决问题、办理业务、必要时找到人工 |
| 人工客服 | 企业坐席 | 接收转接、处理会话、查看用户与历史 |
| 知识运营 | 维护 FAQ、文档切片与质量 | 内容准确、更新及时、可追溯 |
| 系统管理员 | 配置模型、Prompt、风控与账号权限 | 稳定、安全、可审计 |
| 数据/运营分析 | 查看统计与报表 | 漏斗、满意度、热点问题 |

同一自然人可兼任多角色，由权限系统分配。

### 2.3 业务约束

- 主数据与权限以 **Spring Boot + MyBatis-Plus + MySQL** 为权威存储与事务边界。
- 大模型调用、LangChain 编排、向量化与检索以 **FastAPI + Python** 提供独立服务，通过 HTTP/gRPC 与 Java 网关或业务服务通信。
- 前端 **Vue 3 + Element Plus**，与后端通过 RESTful API 及 WebSocket（若采用实时会话推送）交互。

### 2.4 假设与依赖

- 已具备或可申请大模型 API、Embedding API 及向量存储（如 MySQL 8 向量扩展、Milvus、PGVector 等，具体在详细设计阶段确定）。
- 网络可访问模型服务；生产环境需考虑熔断、重试与降级话术。

## 4. 非功能需求

### 4.1 性能

- 常规问答接口：在依赖模型 SLA 前提下，定义 P95 延迟目标（例如 P95 小于 8 秒，以实际模型为准，在详细设计书中量化）。
- 管理端列表分页查询响应在数据量万级内 P95 小于 1 秒（不含复杂分析报表）。

### 4.2 可用性与可靠性

- 核心服务可用性目标建议 ≥ 99.5%（按业务要求可调整）。
- 大模型服务故障时：降级为固定话术 + 转人工引导，避免空白响应。

### 4.3 安全

- 全站 HTTPS；身份认证与鉴权；敏感字段加密存储与传输。
- 防止越权访问他人会话与知识库数据。
- 日志脱敏：手机号、证件号等按规范处理。

### 4.4 可维护性

- Java 与 Python 服务职责清晰，接口契约（OpenAPI/Swagger）版本化。
- 配置与代码分离；关键业务参数可后台调整。

### 4.5 合规与审计

- 对话与操作可追溯；保留周期可配置并符合当地法律要求。

## 5. 技术架构概要（与选型对齐）

| 层次 | 技术 | 职责 |
|------|------|------|
| 前端 | Vue 3、Element Plus | 用户端与管理端 UI |
| 网关/业务 | Spring Boot、MyBatis-Plus、MySQL | 用户与会话、权限、知识元数据、配置、统计落库、对外 API |
| AI 服务 | FastAPI、Python、LangChain | 编排、RAG、调用 LLM/Embedding、向量检索编排 |
| 数据 | MySQL | 业务数据；向量存储方案在详细设计中选定 |

**集成方式：** Java 服务调用 Python AI 服务完成问答链路；异步任务（如批量建索引）可通过消息队列或定时任务触发（实现阶段确定）。

## 6. 外部接口需求（概要）

- **用户端 API：** 注册、登录、退出、凭证刷新（若采用）、会话、消息、评价、转人工等。
- **管理端 API：** 管理员登录、退出、权限校验；知识库、监控、Prompt、模型、敏感词、报表。
- **AI 服务 API：** 对话（含 RAG）、Embedding、重建索引、健康检查。
- **第三方：** 大模型与 Embedding 提供商 HTTP API；可选业务系统 Webhook/REST。

详细字段与错误码在《接口设计说明书》中展开。
